# CBTI 部署包 — 线上卡顿与并发问题分析

本文档在通读 `app.py` 与主要模板逻辑的前提下，从**架构选型、热点路径、前端行为**三方面归纳可能导致「卡顿」「高并发下异常」的原因，并给出改进方向（不强制改代码，便于你按优先级排期）。

---

## 1. 请求与并发模型简述

- 生产环境通过 **Gunicorn** 启动，配置为 **2 个 worker 进程**，每个 worker **4 线程**（见 `Dockerfile` / `Procfile`）。
- 数据库为 **单文件 SQLite**，写操作在任意时刻本质上需要**串行化**（即使开启 WAL，高并发写仍会排队）。
- 每个 HTTP 请求在 `get_db()` 中**新建**数据库连接，并在连接上重复设置 WAL / busy_timeout。

在此模型下，**并发用户数上升**时，瓶颈往往出现在：**SQLite 锁等待**、**整表加载的管理接口**、**单页体积与外部依赖**。

---

## 2. 后端：高概率问题点

### 2.1 `load_records()` 全表加载（严重度：高）

实现逻辑（概念上）：`SELECT * FROM records ORDER BY id ASC` → 逐行 `json.loads` → 组装为 Python 列表。

被以下路径使用：

- **`GET /admin`**：先加载**全部**记录，再在内存里做区域/部门/结果统计，最后才切片分页（每页 150 条）。
- **`GET /api/data`**：返回完整 JSON 列表。
- **`GET /api/export/csv`**：遍历全部记录写 CSV。

**影响**：

- 记录数从几百增长到几千、上万时，**单次请求的 CPU、内存、响应时间**线性上升；管理页打开或导出时容易造成**长请求占用 worker 线程**。
- 与 Gunicorn 线程数有限结合，表现为：**其他用户提交/查询变慢甚至超时**（排队）。

**改进方向**：SQL 层分页（`LIMIT/OFFSET` 或 keyset）、统计用 `GROUP BY` 聚合、导出用流式游标或离线任务；大数据量时迁移 **PostgreSQL / MySQL**。

### 2.2 SQLite 写入与锁（严重度：高，并发场景核心）

- `/api/submit` 每次提交执行 `INSERT` 并 `commit`。
- 多用户同时提交时，多个连接竞争写锁；`busy_timeout=5000` 表示最多可**阻塞约 5 秒**等待锁释放。
- **症状**：用户感知为点击「查看结果」后提交阶段偶发**长时间无反馈**或网关超时（Gunicorn `timeout 30`）。

**改进方向**：合并写请求队列、使用更适合高并发的数据库；若保留 SQLite，可减少事务持有时间、避免在同进程内长事务与写交叉。

### 2.3 每个连接重复执行 PRAGMA（严重度：低～中）

每次 `get_db()` 都执行 `PRAGMA journal_mode=WAL` 等。连接频繁创建时，有一定额外开销；通常不如全表加载明显，但在**极短请求极高 QPS** 时仍属可优化点（例如在进程启动时设一次，或使用连接池且谨慎处理 SQLite 与多线程的边界）。

### 2.4 启动时 `init_db()` / `migrate_db()`（严重度：低）

模块 import 时每个 **worker 进程**都会执行。一般 `CREATE TABLE IF NOT EXISTS` 与一次性的 `ALTER` 可接受；若多 worker 同时首次建库，极少数平台可能出现文件锁竞争，通常短暂。

### 2.5 压缩中间件 `flask-compress`（严重度：中，与数据量相关）

对较大 JSON/HTML 响应做压缩会消耗 **CPU**。当 `/api/data` 或管理页响应体很大时，压缩会放大延迟，并与 2.1 叠加。

**改进方向**：对超大列表接口关闭压缩或改为分页小响应。

### 2.6 管理鉴权为 Session 重定向（严重度：低，体验类）

未登录访问 `/api/data` 等会 **302 到登录页 HTML**，若误用脚本/前后端分离方式调 API，可能被误判为「接口卡顿或格式错误」。非性能主因，但影响联调与监控。

## 3. 前端：卡顿与「伪并发」问题

### 3.1 首屏等待 `window.onload`（严重度：中高）

`index.html` 末尾逻辑：先显示 loading，再在约 200ms 后等待 **`window.onload`**（含图片等静态资源）才显示主内容。

**影响**：

- 若 `/static/*.jpg` **缺失、慢速 CDN、或任意一张图挂死**，会拖慢甚至阻塞首屏展示。
- 当前仓库快照中**未见 `static/` 目录**，部署若未同步图片，浏览器对失败图片的行为仍可能参与 onload 时序；整体原则是：**首屏与测评不应强依赖所有图片加载完成**。

**改进方向**：主内容用 `DOMContentLoaded` 显示；图片 `lazy` 或异步加载；loading 仅针对关键 CSS。

### 3.2 单 HTML 体积巨大（严重度：中）

`index.html` 内嵌完整题库、人格文案、矩阵与大量 CSS。首次下载与解析耗时长，在弱网或移动端更明显。

**改进方向**：题库拆 JSON 按需加载、压缩模板、HTTP/2 多路复用、CDN。

### 3.3 第三方二维码 API（严重度：中）

使用 `api.qrserver.com` 生成二维码。外网延迟、DNS、防火墙或该站限流都会导致**阻塞渲染或长时间 pending**（尤其若图片在 onload 关键路径上）。

**改进方向**：服务端生成二维码、或换自有静态资源、失败降级不阻塞主流程。

### 3.4 `/health` 保活（严重度：低～中）

测评页与管理页均约 **每 4 分钟** 请求一次 `/health`。大量用户长时间挂页时，会产生**额外 QPS**；对 SQLite 单实例而言，`/health` 本身不重，但若前面已有线程耗尽，仍占连接与调度。

**改进方向**：仅管理端保活、或改用平台原生 keep-alive/付费常驻实例。

### 3.5 提交无 UI 防重复（严重度：中）

`submitQuiz` 在发起 `fetch` 后立即 `showResult`，**未禁用按钮、未全局锁**。用户快速连点或网络重试可能导致**重复 INSERT**（同一用户多条记录），加重数据库写压力与统计噪声。

**改进方向**：提交中禁用按钮、幂等键（Idempotency-Key）、后端唯一约束（需业务规则，例如「同工号一段时间仅一条」）。

### 3.6 提交重试逻辑与结果页时序（严重度：中，体验/数据）

前端对 5xx 有重试，但**结果页不等待提交成功**即展示；若最终提交失败，用户看到的结果与库中不一致。易引发「并发下丢数据」的投诉，实为**前后端状态不同步**。

## 4. 并发与一致性层面的其他风险

| 问题 | 说明 |
|------|------|
| **同人多条记录** | 每次测评都是新 `INSERT`；`/api/query` 取 `ORDER BY id DESC LIMIT 1`。**无**「未完成会话」或「覆盖上次」语义，业务上是否接受需产品确认。 |
| **SQLite 与多机部署** | 若水平扩展多台容器各挂独立文件系统，则**数据不共享**；若共享 NFS 上 SQLite，极易损坏与锁问题，**不推荐**。 |
| **清空/删除与正在写入** | 管理端 `DELETE`/清空与用户提交并发时，依赖 SQLite 锁；一般安全但延迟会尖刺。 |

---

## 5. 问题定位建议（线上）

1. **区分慢在浏览器还是服务端**：Chrome 性能面板看 TTFB、下载 HTML 体积、图片与第三方域名 pending 时间。
2. **服务端**：对 `/api/submit`、`/admin`、`/api/data` 打日志或 APM，看 P95/P99 与是否贴近 **5000ms**（busy_timeout）或 **30s**（Gunicorn timeout）。
3. **数据库**：`records` 行数；若过万，优先验证 **2.1 全表加载** 是否为首要矛盾。
4. **压测**：用脚本模拟并发 `POST /api/submit`，观察 SQLite 锁等待与错误率。

---

## 6. 优先级建议（摘要）

| 优先级 | 方向 |
|--------|------|
| P0 | 管理端与 `/api/data` 改为**数据库分页与聚合查询**，避免每次拉全表 |
| P0 | 评估数据量与并发，计划迁移 **Server 型数据库** 或接受 SQLite 的写吞吐上限 |
| P1 | 首屏不依赖 `window.onload` 全资源；二维码失败不阻塞 |
| P1 | 提交防重复 + 可选幂等 |
| P2 | 减少 PRAGMA 重复、大响应压缩策略、保活策略收敛 |
