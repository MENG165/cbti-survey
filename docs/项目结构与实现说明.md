# CBTI 职场人格鉴定所 — 项目结构与实现说明

本文档面向需要部署、二次开发或排查问题的读者，说明本仓库的目录结构、运行时架构、前后端职责与数据流。

---

## 1. 项目定位

本仓库是一个**可直接部署的 Flask 单应用包**，产品名为「CBTI 职场人格鉴定所」（从代码注释与页面标题可知为职场情境人格测评，非医学 CBT-I 失眠治疗）。功能包括：

- 用户在前端完成 **21 道选择题**，本地计算与 10 种「职场人格」模板的 **汉明距离（代码中写作 Haming）** 匹配；
- 将测评结果与答题明细写入 **SQLite**（`data.db`）；
- 提供 **管理后台**（会话登录）用于列表、统计、导出 CSV、单条编辑/删除与清空数据。

## 2. 目录与文件结构

```
CBTI_v3_server_deploy/
├── app.py                 # 唯一后端入口：Flask 应用、路由、数据库访问
├── requirements.txt       # Python 依赖：Flask、Gunicorn、flask-compress
├── Dockerfile             # 基于 python:3.11-slim，Gunicorn 启动
├── Procfile               # 与 Dockerfile 中类似的 Gunicorn 命令（如 Heroku/Railway）
├── data.db                # SQLite 数据库（运行时生成/增长，部署时需可写卷）
├── templates/
│   ├── index.html         # 测评主站：单页 HTML + 内联 CSS/JS + 题库与文案数据
│   ├── admin.html         # 管理后台列表与操作
│   └── admin_login.html   # 管理员登录页
└── docs/                  # 说明文档（本文件等）
```

说明：

- 前端页面通过 Flask `render_template` 渲染，**无独立前端构建步骤**。
- 页面中引用 `/static/*.jpg` 等人格配图；**当前仓库快照中未包含 `static/` 目录**，若线上 404 会影响图片展示与 `window.onload` 行为（详见性能分析文档）。

## 3. 运行时与依赖

| 组件 | 作用 |
|------|------|
| **Flask 3.x** | Web 框架 |
| **Gunicorn** | WSGI 服务器；`Dockerfile`/`Procfile` 中为 `2 workers × 4 threads` |
| **flask-compress** | 对响应做 Gzip/Brotli 压缩 |
| **sqlite3**（标准库） | 持久化 |

启动命令（摘自 `Dockerfile`）：

```text
gunicorn --bind 0.0.0.0:8080 --workers 2 --threads 4 --timeout 30 --graceful-timeout 15 app:app
```

本地开发可直接 `python app.py`，默认 `debug=True`，监听 `7860`（与生产端口不一致，属开发用途）。

## 4. 后端实现（`app.py`）

### 4.1 配置与安全相关

- `SECRET_KEY`：取自环境变量，默认弱密钥（部署应覆盖）。
- `ADMIN_PASSWORD`：管理后台密码，环境变量 `ADMIN_PASSWORD`，默认 `admin123`。
- 登录逻辑中除环境变量密码外，还存在**硬编码备用密码** `cbti2024`（与 `ADMIN_PASSWORD` 二选一即可登录），生产环境建议移除或轮换。
- Session：`SameSite=Lax`、`HttpOnly`；`SESSION_COOKIE_SECURE` 默认为 `False`（HTTPS 站点宜改为 `True`）。

### 4.2 数据库

- 路径：`BASE_DIR/data.db`。
- **启动时**执行 `init_db()` 建表、`migrate_db()` 尝试增加 `region` 列（已存在则忽略异常）。
- `get_db()`：每次调用新建连接，`timeout=10`，`row_factory=sqlite3.Row`，并对每个连接执行：
  - `PRAGMA journal_mode=WAL`
  - `PRAGMA busy_timeout=5000`（5 秒内遇锁等待）

表 `records` 主要字段：

| 字段 | 含义 |
|------|------|
| `id` | 自增主键 |
| `name`, `dept`, `empId`, `region` | 用户填报信息 |
| `result` | 匹配到的人格名称（字符串） |
| `haming` | 与标准模式的汉明距离（数值越小越接近） |
| `dims` | JSON：展示用人格三维度（列表） |
| `answers` | JSON：每题选项键值 |
| `time` | 提交时间（东八区格式字符串） |

### 4.3 路由一览

| 方法 | 路径 | 鉴权 | 行为摘要 |
|------|------|------|----------|
| GET | `/` | 无 | 渲染测评首页 `index.html` |
| POST | `/api/submit` | 无 | JSON 校验姓名/工号/区域后 `INSERT` |
| POST | `/api/query` | 无 | 按姓名+工号查**最新一条**记录 |
| GET | `/admin/login` | 无 | 登录表单 |
| POST | `/admin/login` | 无 | 校验密码后写 session |
| GET | `/admin/logout` | 无 | 清除 session |
| GET | `/admin` | 需登录 | 加载**全部**记录，内存中统计与分页（每页 150 条）后渲染 |
| GET | `/api/data` | 需登录 | JSON 返回**全部**记录 |
| GET | `/api/export/csv` | 需登录 | 导出全部记录为 UTF-8 BOM CSV |
| POST | `/api/update/<id>` | 需登录 | 更新姓名、工号、区域、结果 |
| POST | `/api/delete/<id>` | 需登录 | 删除一条 |
| POST | `/api/clear` | 需登录 | 清空表并重置自增序列 |
| GET | `/health` | 无 | `{"status":"ok"}` |

管理鉴权：`admin_required` 装饰器检查 `session['admin_logged_in']`，未登录则重定向到登录页（**非 JSON 401**，前端若直接调 API 会得到 HTML）。

## 5. 前端实现（`templates/index.html`）

### 5.1 页面结构

单页内包含：

1. **开屏**：品牌文案、说明、表单（姓名、二级部门、区域下拉/其他区域、工号）。
2. **答题屏**：21 题逐题展示，进度条，选项 A/B/C，上一题/下一题；键盘 1/2/3 快捷选择。
3. **结果屏**：人格名称、文案解析、三维度条形图、匹配度说明、二维码分享、重新测试、查看全部 10 种人格弹窗等。
4. **查询弹窗**：姓名+工号调用 `/api/query` 展示最近一次结果摘要。

### 5.2 数据与算法（均在浏览器内完成）

- 脚本内嵌大型常量 `DATA`，包括：
  - `questions`：21 道题的 `id`、题干、`options`（key + text）。
  - `patterns`：10 种人格各自 21 个选项的标准答案序列。
  - `matrix`：每题每选项对三个维度的分值贡献（用于展示「用户维度累计」，**不用于人格匹配**）。
  - `personalityTexts` / `personalityDims`：结果页与详情弹窗的 Markdown 风格文案与固定维度展示值。

**匹配逻辑**（`submitQuiz`）：

1. 按题号顺序收集用户选项，得到长度 21 的 `choices`。
2. 对每种人格的标准 `pattern` 计算汉明距离（不同选项的题数）。
3. 取距离**最小**的人格名作为 `result`（代码注释为「纯模式匹配」）。
4. 将 `PERSONALITY_DIMS[displayName]` 作为 `dims` 与 `userDims`（计算值）等一并 POST 到 `/api/submit`。

注意：提交 payload 中含 `userDims`，但后端 `save_record` **未写入数据库列**，仅保存 `dims` 与 `answers` 等；若将来要做「用户真实维度」分析，需扩展表结构。

### 5.3 与后端的交互

- `fetch('/api/submit', { method: 'POST', body: JSON.stringify(payload) })`：带指数退避的最多 3 次重试（主要针对 5xx，注释写明应对冷启动 502）。
- `fetch('/api/query', ...)`：查询历史结果。
- `fetch('/health', ...)`：**保活**：页面加载后立即请求一次，之后每 **4 分钟** 再请求（与 `admin.html` 中逻辑类似，用于 PaaS 冷启动场景）。

### 5.4 其他前端行为

- 二维码图片使用第三方服务 `https://api.qrserver.com/v1/create-qr-code/...`。
- 页面底部有「加载动画」脚本：在 `window.onload`（**含图片等资源加载完成**）后再显示主内容，可能影响首屏可交互时间。

## 6. 管理后台（`templates/admin.html`）

- 服务端注入：`records`（当前页）、`total`、`page`、`total_pages`、`region_stats` / `dept_stats` / `result_stats`。
- 表格支持列显示切换、客户端排序。
- 通过 `fetch` 调用 `/api/delete/:id`、`/api/clear` 等，成功后 `location.reload()`。
- 同样包含每 4 分钟 `/health` 保活脚本。

## 8. 部署要点（与代码一致的部分）

- 容器内对 `/app` 做了 `chmod -R 777`，保证 `data.db` 可创建与写入。
- 暴露端口：**8080**（与 `app.run` 的 7860 不同）。
- 生产依赖 Gunicorn 的 **多线程 worker**（`gthread` 未在命令行显式写出时，默认 sync worker + `--threads` 的行为以 Gunicorn 文档为准；当前命令使用 `--threads 4`，需使用支持线程的 worker 类型，实际以运行环境 Gunicorn 版本为准）。
